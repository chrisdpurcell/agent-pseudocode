## Last response

I’d propose ADR 0003 as: **“Govern durable repo documentation with standard frontmatter; keep operational and harness files out of scope.”**

Draft convention for your review:

1. Keep `markdown-frontmatter` on schema `1.1`, and do not bundle this with a `project-standards` major-version upgrade. This repo is currently pinned to `standards_version: "v3"` in [.project-standards.yml](/home/chris/projects/agent-pseudocode/.project-standards.yml:1).

2. Managed scope should be broad, but not literally every Markdown file:
   - Include: `docs/**/*.md`, `products/**/*.md`, `hooks/**/*.md`.
   - Exclude: root `README.md`, `TODO.md`, `STATUS.md`, `CHANGELOG.md`, `CLAUDE.md`, `AGENTS.md`, `.claude/**`, `.agents/**`, `.codex/**`, `.github/**`, `docs/handoff/**`, `tests/fixtures/**`, `node_modules/**`, and `**/*.template.md`.
   - Reason: root and handoff files are operational surfaces; skill/harness files often have their own metadata contract; fixtures must stay fixtures.

3. Use the **standard profile**, not the minimal profile, for every managed doc: required fields plus `reviewed`, `owner`, `consumer`, `source`, `confidence`, `visibility`, and `license`.

4. Doc type mapping:
   - Directory/index READMEs: `doc_type: 'index'`
   - `docs/adr/adr-*.md`: `adr`
   - `docs/reference/**`: `reference`
   - procedural install/use/operator docs: `runbook`
   - explanatory docs like mental models or concepts: `concept`
   - roadmap and `docs/superpowers/plans/**`: `plan`
   - `docs/reviews/**`: `research`
   - templates remain excluded unless converted to governed examples.

5. IDs:
   - Ordinary docs use `{doc_type}-{base36-6}-{frozen-slug}`.
   - ADRs keep the current repo-specific ADR form: `adr-NNNN-agent-pseudocode-short-title`.
   - Tokens are generated by tooling/script, never invented by hand.

6. Relationship fields (`related`, `source`, `supersedes`, etc.) use repo-root-relative paths with extensions, not bare IDs, absolute paths, or section anchors.

7. Migration should be gated:
   - First validate the current narrow `docs/adr/**` scope.
   - Add frontmatter mechanically to eligible files while config remains narrow.
   - Review the generated mapping, especially `doc_type`, `status`, and `visibility`.
   - Only then widen `.project-standards.yml`.
   - Do not convert `docs/handoff/bugs/*.md`; their frontmatter feeds the handoff bug index and is a separate local schema.
